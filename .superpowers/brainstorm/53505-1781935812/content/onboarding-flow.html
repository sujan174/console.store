Onboarding — phone as primary key · device-flow auth

Terminal starts it, phone finishes it. Phone number is the account key across all devices.

① First link — terminal → phone

ssh console.store→ terminal shows QR + short link→ 📱 open on phone→ Swiggy app / OTP authorizes→ broker stores token→ phone = account key
Swiggy OAuth doubles as phone proof — no separate console.store OTP needed. No SMS sent on first link.

② Same device, returning

ssh console.store→ pubkey bound ✓→ straight in
Until Swiggy token expires (5 days).

③ New device, existing phone

ssh console.store→ enter phone +91…→ 📱 tap-link sent to phone→ device bound→ in — reuses stored Swiggy token
No Swiggy re-OAuth unless token already expired.

④ Token expiry — every 5 days

next order→ 📱 "tap to re-authorize Swiggy"→ one tap, done
Unavoidable (no refresh tokens in Swiggy v1) — but now it's a single phone tap, not a desktop dance.
Pattern: OAuth 2.1 Device Authorization Grant (RFC 8628) + phone-keyed accounts. Backend holds PKCE verifier & Swiggy token; SSH pubkeys are per-device convenience creds bound after phone verification.
